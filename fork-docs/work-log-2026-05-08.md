# 작업 로그 — 2026-05-08

## 1. 개요 / Goal

이 fork 의 목표는 `codex` CLI 의 모든 LLM 호출을 **별도 플래그 / 별도 설정 파일 없이** 사내 nginx-fronted Ollama 호환 게이트웨이 (`http://10.250.121.100/v1`, 모델 `qwen3.5-122b`, Bearer 인증 `OLLAMA_API_KEY`) 로 라우팅하는 것이다. 게이트웨이는 `/v1/chat/completions` 만 지원하고 `/v1/responses` 는 노출하지 않으므로, 디폴트 라우팅 변경에 더해 **upstream 이 한차례 제거했던 Chat Completions wire API 경로를 복원**하는 것이 핵심 작업이다. 기준 설계 문서는 `fork-docs/ollama-migration.md`.

## 2. 변경 파일별 정리

### 2.1 Provider 라우팅 / 디폴트

- `codex-rs/core/src/config/mod.rs:2634` — 디폴트 `model_provider_id` 를 `"openai"` 에서 `OLLAMA_OSS_PROVIDER_ID` (`"ollama"`) 로 전환. 사용자가 `~/.codex/config.toml` 에 아무 것도 적지 않아도 ollama provider 가 선택된다. `OLLAMA_OSS_PROVIDER_ID` import 는 `codex-rs/core/src/config/mod.rs:78`.
- `codex-rs/model-provider-info/src/lib.rs:415-425` — 빌트인 ollama provider 를 `WireApi::Chat` 로 생성하고 `env_key = "OLLAMA_API_KEY"` 를 박는다. lmstudio provider 는 기존 `WireApi::Responses` 그대로 (`lib.rs:435-438`). `env_key` 가 설정되어 있으므로 `codex-api` 클라이언트가 자동으로 `Authorization: Bearer $OLLAMA_API_KEY` 를 붙인다.

### 2.2 WireApi::Chat 부활

upstream 은 한 시점에 Chat Completions 분기를 통째로 제거했는데, 이 fork 는 게이트웨이 호환을 위해 `Chat` variant 와 그 분기 처리를 살려야 한다.

- `codex-rs/model-provider-info/src/lib.rs:47-82` — `WireApi` enum 에 `Chat` variant 추가. `Display` / `Deserialize` 양쪽에 `"chat"` 매핑을 다시 박았다.
- `codex-rs/core/src/client.rs:1507-1556` — `stream()` 의 wire-api 분기에서 `WireApi::Chat` 매치 암을 추가, `stream_chat_completions_api` 로 디스패치.
- `codex-rs/core/src/client.rs:1566+` — `stream_chat_completions_api` 헬퍼. `output_schema` 는 거부, `effort` / `summary` / `service_tier` / `previous_response_id` 는 무시한다 (Chat Completions 에 등가물 없음).
- `codex-rs/codex-api/src/endpoint/chat.rs`, `codex-rs/codex-api/src/requests/chat.rs`, `codex-rs/codex-api/src/sse/chat.rs` — 클라이언트 / 요청 빌더 / SSE 파서를 history `d2394a2494^` 에서 끌어와 다시 넣었다. `codex-api` mod 트리는 `endpoint/mod.rs`, `requests/mod.rs`, `sse/mod.rs`, `lib.rs` 에서 `chat` 모듈을 다시 export.
- `codex-rs/tools/src/tool_spec.rs:174-200` — `create_tools_json_for_chat_completions_api`. Responses API 용 tool JSON 을 `{"type":"function","name":<n>,"function":{...}}` 모양으로 다시 감싼다. `function` 외 타입 (web_search, image_generation 등) 은 Chat Completions 등가물이 없어 drop.

### 2.3 readiness 흐름 — `--oss` 없이도 ollama 진입

기본이 ollama 가 되었으므로 `--oss` 옵트인 게이트가 의미를 잃었다.

- `codex-rs/exec/src/lib.rs:577-585` — 트리거를 `cli.oss == true` 가 아니라 `config.model_provider_id ∈ {ollama, lmstudio}` 로 변경. `ensure_oss_provider_ready` 호출 위치는 그대로.
- `codex-rs/tui/src/lib.rs:1010-1015` — 동일 패턴 (`model_provider_id` 매치).

### 2.4 원격 게이트웨이 호환 보강

원격 OpenAI 호환 게이트웨이는 Ollama admin endpoint (`/api/tags`, `/api/version`) 를 노출하지 않거나 인증으로 가린다. readiness 단계에서 fatal 이 떨어지지 않게 두 군데를 수정.

- `codex-rs/ollama/src/lib.rs:16` — `DEFAULT_OSS_MODEL = "qwen3.5-122b"` (upstream 의 `gpt-oss:20b` 에서 변경).
- `codex-rs/ollama/src/lib.rs:37-56` — `ensure_oss_ready` 가 `fetch_models()` 빈 리스트를 만나면 자동 pull 을 skip 하고 사용자 모델 선택을 신뢰. 실제 라우팅 실패는 첫 chat completions 호출에서 표면화된다.
- `codex-rs/ollama/src/lib.rs:75-93` — `ensure_responses_supported` 가 `provider.wire_api == WireApi::Chat` 이면 즉시 `Ok(())` 로 short-circuit. `/api/version` 미노출 게이트웨이에서 fatal 회피.
- `codex-rs/ollama/src/client.rs:88-106` — `probe_server` 가 모든 HTTP 응답 (401/403/404 포함) 을 "도달 OK" 로 해석. transport-level connection error 만 fatal. 인증 가드된 `/v1/models` 가 401 을 던져도 codex 는 진행.

### 2.5 chat completions 어댑터의 role 매핑 (오늘, 2026-05-08)

게이트웨이가 표준 OpenAI chat completions 의 `role` enum (`system` / `user` / `assistant` / `tool`) 만 받는다. Codex 내부는 instructions / goals / tasks 를 전달할 때 `role = "developer"` 를 사용한다 (Responses API 컨벤션).

- 1차 시도: `developer → system` 매핑. 게이트웨이 응답 `System message must be at the beginning.` (게이트웨이 정책상 leading system 메시지 1개만 허용).
- 최종: `developer → user` 매핑.

`codex-rs/codex-api/src/requests/chat.rs:198`:

```rust
let outbound_role = if role == "developer" { "user" } else { role };
let mut msg = json!({"role": outbound_role, "content": content_value});
```

같은 빌더 안의 `last_emitted_role` 추적 (`requests/chat.rs:64-96`) 은 원래 role 그대로 두었다. 이 추적은 trailing user 메시지 합성 로직에 쓰이는데 원본 role 기준으로 동작해야 정확하므로 의도적으로 미변경.

### 2.7 외부 JSON config 주입 (오늘, 2026-05-08, opencode 호환)

`baseURL` / `apiKey` / `model` 세 값을 외부 JSON 파일에서 주입받아 환경변수에 export 하는 fork-only 로더.

- `codex-rs/ollama/src/fork_config.rs` — 신설. `apply_fork_config_to_env()` 가 `$CODEX_FORK_CONFIG` 또는 `$HOME/.codex/codex-fork.json` 을 읽어, `baseURL` → `CODEX_OSS_BASE_URL`, `apiKey` → `OLLAMA_API_KEY`, `model` → `CODEX_OSS_MODEL` 환경변수에 쏟는다. 파일 부재 / 파싱 실패는 silent skip + `tracing::warn`. JSON 의 값이 셸 env 보다 우선 (loader 가 unconditional `set_var`).

- `codex-rs/ollama/src/lib.rs:1-9` — 모듈 추가 + `apply_fork_config_to_env`, `fork_model_override` re-export.
- `codex-rs/utils/oss/src/lib.rs:7-8` — codex-utils-oss 에서 다시 re-export. exec / tui / cli 가 동일 dep 으로 접근.
- `codex-rs/cli/src/main.rs:745-749` — `cli_main` 진입 첫줄에서 `codex_utils_oss::apply_fork_config_to_env()` 호출. 모든 subcommand 디스패치 이전이라 codex / codex exec / codex mcp-server 등 진입점에서 모두 적용.
- `codex-rs/exec/src/lib.rs` , `codex-rs/tui/src/lib.rs` — model 해석 분기에서 `--oss` 도 아니고 CLI `-m` 도 없을 때 `codex_utils_oss::fork_model_override()` (= `CODEX_OSS_MODEL` 환경변수) 를 fallback 으로 끼워 ConfigOverrides::model 에 반영.
- `codex-rs/cli/Cargo.toml` — `codex-utils-oss` dep 추가. `codex-rs/ollama/Cargo.toml` — `serde = { workspace = true, features = ["derive"] }` 추가.

JSON 스키마 예시 (`~/.codex/codex-fork.json`):


end-to-end 검증: 셸 env 를 모두 unset 한 상태에서 위 파일만 두고 `cargo run --bin codex -- exec "say pong"` 통과 확인. `CODEX_FORK_CONFIG=/tmp/myfork.json` 으로 임의 경로 지정 시나리오도 동일하게 통과.

## 3. 검증

오늘 시점 end-to-end 검증:

```bash
# 방법 A — 셸 환경변수
export CODEX_OSS_BASE_URL='http://10.250.121.100/v1'
export OLLAMA_API_KEY='<122b key>'    # sk-davis-122b-... 토큰
cargo run --bin codex -- exec "say pong"

# 방법 B — opencode 식 외부 JSON (셸 env 0개)
cat > ~/.codex/codex-fork.json <<'JSON'
{ "baseURL": "http://10.250.121.100/v1",
  "apiKey":  "<122b key>",
  "model":   "qwen3.5-122b" }
JSON
cargo run --bin codex -- exec "say pong"
```

- `Authorization: Bearer <122b key>` 헤더로 `POST http://10.250.121.100/v1/chat/completions` 호출
- 응답 본문에 `pong` 도착, 정상 종료
- `--oss` 플래그 / `~/.codex/config.toml` 항목 없이 디폴트 라우팅으로만 동작 확인
- 방법 B 도 동일하게 통과 (env 만으로도, JSON 만으로도, 그리고 둘 다 있을 때 JSON 이 우선)

`developer → user` 매핑 변경 직전까지는 게이트웨이가 `400 Unexpected message role.` (또는 `developer → system` 1차 시도 시 `System message must be at the beginning.`) 로 거부했고, 매핑 적용 후 동일 명령이 통과하는 것으로 회귀 확인.

## 4. 잔존 항목 / 알려진 제한

- **chat completions 테스트 비활성**: `codex-rs/core/tests/chat_completions_payload.rs:7`, `codex-rs/core/tests/chat_completions_sse.rs:4` 모두 `#![cfg(any())]` 로 컴파일에서 제외되어 있다. upstream protocol drift (`ResponseItem::Message` 의 `end_turn` 제거, `ResponseEvent::Completed` 의 필드 이동 등) 때문에 그대로 빌드되지 않아 일단 격리. end-to-end 가 안정화된 후 케이스 단위로 살려야 한다.
- **NUDGE_MODEL_SLUG 무력화**: `codex-rs/tui/src/chatwidget.rs:459` 에 `NUDGE_MODEL_SLUG = "gpt-5.4-mini"` 가 박혀 있다. `models.json` 에서 해당 슬러그를 제거했으므로 picker preset 매칭 (`chatwidget.rs:7176`) 과 nudge 발화 조건 (`chatwidget.rs:2951`) 이 실질적으로 dead path 가 되었다. 동작이 깨지지는 않지만 silent no-op. 이후 Qwen 슬러그 또는 Option 으로 정리 필요.
- **gateway 키 ↔ 모델 강제 매핑**: 게이트웨이는 키마다 사용 가능한 모델을 제약한다. `OLLAMA_API_KEY` 가 122b 키일 때만 `qwen3.5-122b` 가 통과한다. 키와 `models.json` 슬러그가 일치하지 않으면 게이트웨이가 `Invalid API key for model: ...` 로 거부한다.
- **Responses API 경로 미점검**: 이 fork 에선 `WireApi::Chat` 만 실사용하므로 `WireApi::Responses` 분기를 건드린 변경은 회귀 테스트가 빈약하다. lmstudio 는 형식상 살아있지만 fork 운영 환경에서는 검증 대상 아님.
- **`role` 매핑 유실 가능성**: `developer → user` 단방향 매핑이라 어시스턴트가 "user 가 한 발화" 와 "developer instruction" 을 구분할 수 있는 구조적 신호가 사라진다. 현재 instructions 는 system 메시지 1개로 들어가므로 큰 문제는 없으나, 향후 multi-stage developer turn 이 늘어나면 prefix 컨벤션 (예: `[SYSTEM]`) 추가를 고려.
- **upstream rebase 시 충돌 표면**: `WireApi::Chat`, `codex-api/src/{endpoint,requests,sse}/chat.rs`, `tools/src/tool_spec.rs` 의 chat completions 헬퍼는 모두 upstream 이 의도적으로 제거한 코드 경로다. rebase 시 large conflict 가 예상된다.
